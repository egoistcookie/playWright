###标题###[AI-开发过程-熠辉IndieDev] 

20251024：

00:00:00Vibe Coding 的介绍及其局限性

·Vibe Coding 能让初学者使用AI编写代码和构建网站。

·尽管有最强大的大型模型和AI编程工具，但要完成一个完整的前后端商业项目（带有完全装备的管理后台）仍可能感到无从下手。

·讲者将分享他使用AI开发的一个商业项目——网球场地预订小程序，并展示从初始需求到代码开发再到最终上线运营的完整过程。

00:00:32项目开发前的准备工作的重要性

·开发项目的第一个也是最重要的步骤是在编写任何代码之前做好准备。

·最重要的是产品需求文档，这是项目的“设计蓝图”，应从产品角度而非技术角度出发。

·不需要在此阶段讨论具体的技术细节，如使用React或MySQL，这些可以在后期考虑。

00:01:02产品需求文档的具体内容

·需要告诉AI产品具体做什么，包括页面、按钮以及点击后的操作。

·讲者提供了一个他常用的模板，直接给Claude输入，并用简单语言解释需求。

·示例了如何描述一个网球小程序的需求，包括底部的四个按钮及其功能。

00:01:28原型设计的重要性

·文字描述不足以清晰表达想法，需要创建原型使想法可视化。

·可以使用传统工具如Figma或Mockingbot，但现在讲者更倾向于让AI直接生成，例如使用VO或让Claude生成交互式HTML页面。

Q：怎么让ai生成交互式HTML页面？

A： A：

·虽然生成的页面可能看起来不太美观，但功能和工作流程一目了然。

00:01:46原型和需求文档的相互完善过程

·在绘制原型时，可能会发现文档中某些逻辑没有明确思考清楚。

·相反，文档可以指导你完善原型使其更加精致。

·讲者强调在这个阶段投入更多精力，后续开发过程会轻松十倍，也能从根本上避免写出糟糕的代码。

00:02:15前端开发的建议

·前端开发建议首先引入成熟的UI框架，对于小程序推荐使用腾讯官方的TDesign。

·这相当于为AI提供了一套高级的乐高积木，帮助更快更高效地构建漂亮的页面。

·提倡让AI将页面数据提取到单独的JSON文件中，便于未来连接后端接口时替换该文件。

00:02:44后端开发的关键步骤

·后端开发记住一条黄金法则：首先开发登录功能，因为几乎所有后续功能都与当前用户有关。

·在集成关键框架、数据库等技术栈之后，让AI生成一个名为Claude.md的项目规范文档，定义项目的技术架构、编码标准等。

·这相当于为AI设定规则，使其在整个项目中按照要求工作，确保统一的代码风格。

00:03:15功能开发的最佳实践

·绝不能一次性把整个项目丢给AI，那样肯定失败。

·正确的做法是逐个开发和测试功能，例如基于模板让Claude生成包含推荐开发任务计划的提示。

·每个任务都会被分解成多个独立可执行的小任务，每个任务都有清晰的提示可以直接复制粘贴，逐步实现功能。

00:04:01项目上线前的关键步骤

·特别指出这些最后的步骤是玩具项目与商业项目之间的关键区别。

·第一是环境区分，必须至少有两个环境：生产环境和内部测试环境，它们的数据库域完全分离，以防干扰线上用户的实际数据。

·第二是设置网站的元数据和OG图片，这影响产品在社交媒体上的传播效果。

·第三是数据跟踪，了解用户如何与产品互动，哪些按钮被点击最多，哪些页面停留时间最长，这些数据有助于优化产品和做出商业决策。

00:05:01AI开发商业项目的结构化过程总结

·在开发前，采用产品视角精炼需求文档和原型图，这是基础。

·在开发阶段，将主要功能分解为小任务，制定详细的作战计划，逐一完成。

·在开发完成后，需要正确区分环境、宣传材料和数据跟踪点，这是专业性的保证。

8-3 Web与Spring的历程

一切的源头Servlet：Struts2的源头其实也是Servlet。Servlet的作用是接收浏览器传给服务端的请求（request），并将服务端处理完的响应（response）返回给用户的浏览器，浏览器和服务端之间通过http协议进行沟通，其过程是浏览器根据用户的选择将相关信息按http协议报文的规范组装请求的http报文，报文通过网络传输到指定的服务器，服务器通过特定的web容器接收这个报文信息，例如：tomcat，jetty，jboss这样的web容器，web容器会将http报文解析出来，如果是用户请求，最终解析出来的报文信息会用一个request对象存储起来，服务端使用这个request做完相应的处理后，服务端程序将结果信息封装到response对象里，然后将response对象交给web容器，web容器则把这个response对象转变为http协议的报文，并将报文回传给浏览器，浏览器最后解析这个响应报文，将最终结果展示给用户。

Web容器创造了servlet接口，servlet接口就是开发人员自己实现业务逻辑的地方，程序员开发servlet就好比做填空题，而填空题的语境或者说上下文提示就是由request和response对象，但是javaEE规范里的servlet接口很简单，就三个方法init，service和destory，但是这个接口太笼统，所以规范里还提供了一个HttpServlet类，这个类根据http请求类型提供了doGet，doPost等方法，servlet接口最大的特点就是根据http协议的特点进行定义，因此做servlet开发时候如果使用者对http协议特点不是特别熟悉，都会碰到或多或少令人迷惑的问题，特别是碰到一些复杂特殊的请求时候：例如文件上传，返回特殊的文件格式到浏览器，这时候使用servlet开发就不是很方便了，servlet开发还有个问题可能大家常常被忽视，就是请求的数据的类型转化，http协议传输都是文本形式，到了web容器解析后也是文本类型，如果碰到货币，数字，日期这样的类型需要我们根据实际情况进行转化，如果页面传送的信息非常多，我们就不得不做大量类型转化，这种工作没有什么技术含量，是个体力活而且很容易导致程序错误。同时java的企业开发都是围绕javabean进行，类型转化好的数据还要封装到对应的javabean里，这种转来转去的事情对于项目开发绝对不是什么好事情，所以古老的struts1为这种问题找到了一种解决方案，就是定义了一个DTO对象（数据传输对象），专门负责做这样的事情，不过到了struts2，整个替代servlet的action本身就是一个javabean。

Servlet在MVC开发模式里就是其中C层即控制层，控制层就像俄罗斯的双头鹰（一个头向东看一个头向西看）一样，一个头向M层模型层看，一个头向V层视图层看，模型层也是用java编写的，控制层也属于服务端语言开发，所以M层和C层的沟通没有天然的障碍，但是和V层视图层就不一样了，这是一个跨语言的沟通，对于浏览器，它只懂得html，javascript和css，浏览器是理解不了java这种语言的东西，但是要让服务端的东西能被浏览器理解接受，我们就必须得把服务端的响应信息放到页面里，因此就需要一个技术把java的信息转化到html页面里，这就是javaEE规范里提供了jsp技术，jsp其实是一种服务端技术而非客户端技术，不过它看起来似乎更像html技术，最早的jsp开发里都是直接将java代码写到页面里，这种坏处谁都知道，之后javaEE规范提供了自定义标签技术，使用一种类似html标签的方式来解析java代码，struts2框架提供了一整套完整的自定义标签技术，这似乎听起来不算啥，但是它的作用非凡，因为自定义标签之所以叫自定义就是每个人都可以自己来定义，如果没有一个规范必然产生混乱，而且一套完善的自定义标签是个系统工程，一套完整的自定义标签相当于我们在自己定义一套新的开发语言，做程序的人听到这个一定就会明白开发一套完整的自定义标签的工作量和开发难度都是难以想象的，而且自定义标签都是和控制层紧密相连，其难度又会增加一个维度，所以struts2提供的自定义标签对于业务开发带来的将是质的飞越。

Servlet里还有两个重要的技术：监听器和过滤器，对于监听器在web开发里使用的场景比较少，都是一些十分特别的情况才会使用，大部分web开发里可以忽略它的使用，我们用的最多的监听器可能就是对ServletContext创建和销毁的监听器，ServletContext是整个web应用的全局对象，它和Web应用的生命周期绑定在一起，因此使用这个监听器对Web应用的全局信息进行初始化和销毁操作，例如spring容器的初始化操作。比较有意思的是过滤器，在struts2里有个拦截器，它们的作用相同都是用来拦截请求的，因为拦截器是struts2的特有功能，在struts2里使用拦截器自然比使用过滤器更顺手，其实拦截器所用的技术比过滤器更加先进，因为拦截器使用了反射技术，因此拦截器拦截的面更大，控制请求的能力更强，它能完成的任务也会更加的丰富多彩。

###标题###[Thymeleaf] 

Thymeleaf：一种服务器端模板引擎。允许在HTML中通过特定的属性标签与后端Java代码（如Spring MVC模型）进行数据绑定，语法自然，易于理解和调试。非常适合渲染传统的后端管理页面。它的主要目标是提供一种优雅且高度可维护的模板创建方式。

模板文件可以直接在浏览器中打开，因为它使用的是标准 HTML 结构，标签和属性都是合法的 HTML 语法。

与 Spring MVC 有出色的集成，是 Spring Boot 推荐的模板引擎之一。

在<title>标签中通过th:text语法动态渲染标题内容。

提供了多种表达式语法，用于访问变量、执行运算、处理集合等。

Q：jsp的补丁，一个优雅的自定义标签库？

A：错，Thymeleaf 确实与 JSP 有相似的用途（都是服务器端模板引擎），并且在某些方面可以看作是对 JSP 局限性的改进，但它并非简单的 “补丁” 或仅局限于 “自定义标签库” 的范畴。与 JSP 的核心差异：JSP 本质上是在 HTML 中嵌入 Java 代码（通过 <% %> 等标签），导致模板与业务逻辑耦合度高，且原生模板无法直接在浏览器中预览（必须经过服务器解析）。而 Thymeleaf 采用 “属性驱动” 模式，所有逻辑通过 th:* 等自定义属性实现，完全符合 HTML 标准，模板文件可直接在浏览器中打开查看静态效果，这使得前端与后端协作更顺畅。是超越了标签库的模板引擎，是jsp的替代方案而非补丁。
